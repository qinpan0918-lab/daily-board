- 同时修复了 `getDB()` 中对空数据库的检测逻辑（即使本地有 db 文件也检查用户数）

## 12:01 - 空库覆盖防护修复（第三次根治）✅

### 问题
用户反馈切到4月8号依然没数据。检查发现：
- Render 又重部署了一次 → 数据库清空
- 启动后15秒 `pushToGitHub()` 把**空的0任务数据库**推送到 GitHub → **覆盖了之前的7条任务备份**
- 恶性循环：每次重部署 → 空库 → 覆盖远程 → 数据永久丢失

### 根因
`pushToGitHub()` 缺少安全保护——不区分"有数据的正常备份"和"空库覆盖"，无脑推送。

### 修复内容（server/server.js）
1. **pushToGitHub() 加空数据保护**：
   - 推送前先检查本地 task 数量
   - 如果本地=0但远程>0 → 自动从远程拉取恢复到本地，用恢复后的数据推送
   - 绝不允许空库覆盖有数据的远程备份
2. **拆分 githubPushJSON()**：把实际 API 推送逻辑独立为函数，避免递归问题
3. **启动恢复增强**：
   - 不仅检查 userCount=0，还检查 taskCount=0
   - 增加详细日志输出
   - 延迟首次自动备份从 15s→30s
4. 已推送到 GitHub（commit: 3f6b089），Render 自动重新部署
